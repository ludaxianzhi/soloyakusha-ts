# 系统化项目分析与 QWEN.md 生成流程

当需要全面理解一个新代码库并生成 `QWEN.md`（供未来交互使用的项目上下文文档）时，遵循以下迭代分析方法。

## 第 1 步：初始概览

先做宽扫描，不扎入细节：

1. **列出根目录** — 查看顶层文件和目录结构（`list_directory` root）。
2. **阅读 README** — 了解项目目的、技术栈、快速开始命令。
3. **阅读关键配置**：
   - `package.json` / `Cargo.toml` / `go.mod` / `pyproject.toml` — 依赖、脚本、项目元数据
   - `tsconfig.json` / `Makefile` / `.babelrc` — 编译/构建选项

## 第 2 步：分析入口点和公开 API

- 读取入口文件（如 `index.ts`、`main.ts`、`lib.rs`、`__init__.py`），了解模块的公开 API 和模块依赖。
- 这一步揭示了项目的"骨架"——模块是如何组织的、核心概念有哪些。

## 第 3 步：深度探索模块（迭代式，最多 10+ 轮）

**不要一次性决定读哪些文件**。让每次探索的结果引导下一步：

1. **列举 `src/` 或 `lib/` 子目录**，了解各模块职责。
2. **对最复杂的模块，使用 subagent 或批量列举**获取其中所有文件列表和简短说明。
3. **阅读你觉得最重要的 2-3 个核心文件**（如 provider、main facade、factory）。
4. **遇到设计文档**（`docs/`、`design/`、`ADR-*.md`）——阅读它们。它们通常记录了过去的关键决策和陷阱。
5. **读取 CI/CD 配置**（`.github/workflows/`、`.circleci/`）——了解构建、测试、发布流水线的状态。
6. **查看测试文件的命名模式和数量**——了解测试策略。

**关键原则**：每次阅读后问自己"我还需要了解什么才能理解这个项目？"让答案引导下一步。

## 第 4 步：识别关键约定和陷阱

从代码和文档中提取以下信息：

| 类别 | 寻找什么 |
|------|---------|
| 代码规范 | 模块导出模式、类型命名、路径分隔符风格、测试框架 |
| 配置约定 | 配置文件位置、规范化/验证方式 |
| 已知陷阱 | 已有设计文档中记录的问题（如 id 字段丢失、作用域隔离缺陷） |
| 同步要求 | 修改某个值需要同步多少个位置（如上下文网络阈值） |
| 状态信息 | CI 哪些测试通过/失败、构建是否成功 |

## 第 5 步：生成 QWEN.md

输出文档应包含以下章节：

```markdown
# <项目名> — <一句话描述>

## 项目概述

简要说明：项目目的、核心能力、技术栈、运行时。

## 构建与运行

所有关键命令，按使用频率排列：安装 → 类型检查 → 测试 → 构建 → 运行。
包括已知的失败/警告（如"已知 XX 测试失败"）。

## 项目结构

目录树 + 每个模块的一句话说明。标记出最复杂的模块。

## 架构要点

- 模块依赖关系图（ASCII 或描述性）
- 关键设计决策（如"WebUI 薄层原则"、"单入口导出"）
- 持久化策略（文件/SQLite 位置）
- 隔离/并发模型（如 AsyncLocalStorage）

## 关键约定

表格形式呈现代码规范、配置约定、已知陷阱、同步要求、CI 状态。

## 流程概览（可选）

端到端的数据流，帮助快速理解使用场景。

## 常用命令速查

简单表格：命令 → 用途
```

## 整体原则

- **逐步求精**：先宽后深，每次迭代缩小范围
- **文档优先**：README、设计文档、CI 配置是最高信噪比的信息源
- **记录已知问题**：项目中已有的 Bug、失败测试、未重构的部分——这些是未来开发者的重要上下文
- **保存架构决策**：为什么这样设计比"是什么"更重要，它们在设计文档中

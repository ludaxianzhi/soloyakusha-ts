# 大型工作区打开性能调查与优化

## 代码分析——疑似瓶颈

通过追踪 `TranslationProject.initialize()` 完整调用链，发现以下高开销模式：

### 瓶颈 1：`loadChapterIndexSync()` 每章开闭一次 DB
**路径**：`initialize()` → `loadChapters()` → for 每个章节调用 `loadChapterIndexSync(chapterId)`

`loadChapterIndexSync()` 每次调用都会 `openDatabase()`（创建新连接、执行 WAL PRAGMA、执行全部 `CREATE TABLE IF NOT EXISTS` DDL）、执行一条 SELECT、然后 `close()`。200 章 = 200 次 DB 开闭。

### 瓶颈 2：`ensureChapterLoaded()` 延迟加载触发 N 次 DB 开闭
**路径**：`initialize()` → `initializePipelineQueues()` → 遍历所有 fragment 调用 `getPipelineStepState()` → `getRequiredFragment()` → `getFragmentById()` → `getChapterById()` → `ensureChapterLoaded()` → `loadChapterSync()`

`loadChapterSync()` 每次调用开闭 DB，执行 3 条 SELECT（chapters、fragments、fragment_lines、pipeline_step_states）。200 章 = 200 次 DB 开闭。

### 瓶颈 3：`getProjectSnapshot()` 频繁触发延迟加载
**路径**：前端轮询 → `getSnapshotWithEntries()` → `project.getProjectSnapshot()` → `buildQueueEntrySnapshot()` → `documentManager.getPipelineStepState()` / `getSourceText()` / `getTranslatedText()`

每个条目触发 chapter body 的延迟加载（如果尚未在内存中）。运行时每次 UI 刷新都会触发。

### 瓶颈 4：`initializePipelineQueues()` 每 fragment 调 `getPipelineStepState()`
对已经入队/完成的步骤再次从 fragment pipelineStates 中读取状态，但每次都必须通过 `getRequiredFragment()` 路径，可能触发额外的内存查找链。

使用 `performance.now()` 分别在 `initialize()` 各阶段打点，打印到 stderr（不影响 stdout 输出）。

## 第二阶段：根据测量结果实施优化

### 优化 A：批量加载章节索引

新增 `loadAllChapterIndexesSync()` 方法，一次性读取所有章节索引：

```sql
SELECT c.chapter_id, c.file_path, f.fragment_index, f.hash
FROM chapters c
LEFT JOIN fragments f ON f.chapter_id = c.chapter_id
ORDER BY c.chapter_id, f.fragment_index
```

`loadChapters()` 改为单次 DB 打开，调用 `loadAllChapterIndexesSync()` 后关闭。

### 优化 B：预加载全部章节内容

在 `loadChapters()` 完成后，一次性批量加载所有已持久化章节的完整数据：

新增 `loadAllChaptersSync()` 或改造现有方法，单次 DB 连接中读取全部 chapters + fragments + fragment_lines + pipeline_step_states，hydrate 后写入内存。

### 优化 C：getChapterDescriptors 已足够用

`getRequiredChapterDescriptor()` 在 Workspace 中已优化为优先从 `loadChapterDescriptorsSync()` 读取（已在上一轮重构中实现），此路径已是 O(章节数) 列直读。

### 优化 D（如需要）：Snapshot 缓存

检查 `buildQueueEntrySnapshot()` 中对 `getSourceText()` / `getTranslatedText()` 的调用是否可以被跳过（非运行时不需要实时数据），或缓存到 in-memory。

## 预期效果

- DB 开闭从 O(章节数 × 2) 降为 O(1)
- 章节体批量加载在单事务/单连接中完成
- `loadChapterDescriptorsSync()` 已为列直读（O(章节数)），不需进一步优化

---

## 注意事项

1. **内存占用**：批量加载全部章节到内存会增加内存占用（已有惰性加载机制就是为了省内存），但对于大型工作区打开场景，速度优先。
2. **并发写互斥**：`loadAllChaptersSync()` 是只读操作，不需要进入 `enqueueWrite` 队列。
3. **事务一致性**：批量读不需要事务，但应在同一个 Database 实例中完成。

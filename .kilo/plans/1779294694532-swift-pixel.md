# 工作区页面章节列表性能优化方案

## 问题定位

### 根因：500-1000 章节全部渲染到 DOM

`WorkspaceChaptersTab.tsx` 使用 Ant Design `<Table pagination={false}>`，**所有行同时渲染到 DOM**。Antd Table 每行有大量嵌套组件，500-1000 行 × 8 列 ≈ 4000-8000 个 React 组件实例同时存在，导致：

- **首次渲染**：JS 执行 + DOM 创建耗时过长（可轻易超过 500ms）
- **每次状态更新**（选中、筛选、数据刷新）→ 触发 500-1000 行重渲染
- **交互响应**：点击行触发选中状态更新，React reconciliation 扫描整个 Table 子树

### 次因：5 秒轮询重新获取全部章节

`usePollingTask` 每 5 秒调用 `onRefreshChapters()` → `setChapters(chaptersRes.chapters)` → **即使数据未变，也会产生新数组引用** → 触发整个 WorkspaceView → WorkspaceChaptersTab → ChapterInfoTable → ChapterTableSection 的完整重渲染链。

### 次因：无虚拟滚动库

项目中无 `react-window` / `@tanstack/react-virtual` / `react-virtuoso` 等虚拟滚动依赖，无法只渲染可视区域的 DOM 节点。

---

## 短期解决方案（已实施 ✅）

### 1. 给 Table 开启分页

在 `WorkspaceChaptersTab.tsx:337` 给 `<Table>` 增加分页配置（默认 50 条/页）。

**效果**：DOM 节点从 500-1000 行降到 50 行，大幅削减渲染/重渲染成本。

**额外处理**：由于分页模式下表头全选复选框只勾选当前页，设置了 `hideSelectAll: true` 并在工具栏增加了"全选所有章节"和"取消全选"按钮，操作 `filteredChapters` 全量数组。

### 2. 用 React.memo 包裹 ChapterInfoTable

避免 `WorkspaceChaptersTab` 因非 chapters 状态变化导致树不必要的重渲染。

### 3. 轮询间隔 5s → 15s

减少 3 倍网络请求和 setChapters 触发的重渲染链。

## 编排页面（Kanban）卡顿分析与优化方案

### Kanban 卡顿根因

`ChapterKanbanBoard.tsx` 渲染机制：

1. **所有章节卡片同时渲染**：KanbanColumn 将路线内所有 `chapterIds` 映射为 `<KanbanCard>` 组件，500-1000 章节对应同等数量 DOM 节点
2. **每张卡片附加 `useSortable`（@dnd-kit）**：每个 KanbanCard 调用 `useSortable` 注册 drag & drop 事件绑定和状态追踪，500-1000 个 sortable 实例带来额外 JS 开销
3. **无 React.memo 保护**：`KanbanColumn` 和 `KanbanCard` 都是普通函数组件，当父组件因任何状态变化（如 `activeId` 随拖拽变化、`chapters` 因轮询刷新）重渲染时，全部列和全部卡片同步重渲染
4. **`onOpenPerChapterExport` 内联箭头函数**：每次 `ChapterKanbanBoard` 重渲染都创建新函数引用，传递给每个 `KanbanCard` 的 prop
5. **`menuItems` 数组每个渲染帧重建**：KanbanCard 内固定结构的 `menuItems` 数组每次渲染重新创建
6. **内容密度高**：每张卡片包含进度条、4 个方向按钮、分支按钮、更多下拉等大量交互元素

### 短期方案（可单独实施）

#### 方案 A：React.memo + 稳定回调（推荐，1 天内完成）
- 用 `React.memo` 包裹 `KanbanCard` 和 `KanbanColumn`
- 将 `onOpenPerChapterExport` 内联函数提取为 `useCallback` 稳定引用
- 将 `menuItems` 提取为组件外部常量或 `useMemo`
- 效果：拖拽时只有被拖拽卡片和目标列重渲染，其他列/卡片跳过渲染

#### 方案 B：分页/折叠大路线（可选，0.5 天）
- 当路线内章节数 > N（如 100）时默认折叠，显示"展开全部 N 章节"按钮
- 或使用分页组件（类似列表页）限制每列渲染卡片数
- 效果：大幅削减 DOM 节点

### 长期方案（底层重构）

1. **虚拟滚动路线内卡片**：引入 `@tanstack/react-virtual`，KanbanColumn 内的卡片列表改为虚拟列表，只渲染可视区域的 N 张卡片
2. **服务端分页/增量加载**：`api.getChapters` 支持分页，Kanban 只加载当前视图需要的章节子集
3. **懒加载路线**：路线数量多时（如 >5 条路线），只渲染可视区域内的路线列
4. **数据刷新优化**：将轮询改为 SSE 推送变更通知，避免全量数据轮询

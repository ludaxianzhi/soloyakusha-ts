# 工作区页面章节列表性能优化方案

## 问题定位

### 根因：500-1000 章节全部渲染到 DOM

`WorkspaceChaptersTab.tsx` 使用 Ant Design `<Table pagination={false}>`，**所有行同时渲染到 DOM**。Antd Table 每行有大量嵌套组件，500-1000 行 × 8 列 ≈ 4000-8000 个 React 组件实例同时存在，导致：

- **首次渲染**：JS 执行 + DOM 创建耗时过长（可轻易超过 500ms）
- **每次状态更新**（选中、筛选、数据刷新）→ 触发 500-1000 行重渲染
- **交互响应**：点击行触发选中状态更新，React reconciliation 扫描整个 Table 子树

### 次因：5 秒轮询重新获取全部章节

`usePollingTask` 每 5 秒调用 `onRefreshChapters()` → `setChapters(chaptersRes.chapters)` → **即使数据未变，也会产生新数组引用** → 触发整个 WorkspaceView → WorkspaceChaptersTab → ChapterInfoTable → ChapterTableSection 的完整重渲染链。

### 次因：无虚拟滚动库

项目中无 `react-window` / `@tanstack/react-virtual` / `react-virtuoso` 等虚拟滚动依赖，无法只渲染可视区域的 DOM 节点。

---

## 短期解决方案（立即见效，低风险）

### 1. 给 Table 开启分页（单文件修改，5 分钟）

在 `WorkspaceChaptersTab.tsx:337` 给 `<Table>` 增加分页配置：

```tsx
<Table
  pagination={{
    defaultPageSize: 50,
    showSizeChanger: true,
    pageSizeOptions: ['20', '50', '100', '200'],
    showTotal: (total) => `共 ${total} 章节`,
  }}
  // ...
/>
```

**效果**：DOM 节点从 500-1000 行降到 50 行，大幅削减渲染/重渲染成本。

**影响**：
- +) 即刻修复卡顿
- +) Shift+Click 区间选择仍然在 `filteredChapters` 全量数组上操作，选中功能不受影响
- +) 筛选功能不受影响（筛选计算在 `useMemo` 中，对分页后的 Table 数据源无影响）
- -) 用户需要翻页浏览章节列表，操作体验略有变化。但考虑到 500-1000 章节时分页是合理做法

### 2. 优化 `toolbarActions` 的依赖项

`toolbarActions` 的 `useMemo` 依赖 `[..., selectedChapterIds.length]` 但内部引用了 `selectedChapterIds` 本身。当 `selectedChapterIds` 数组内容变化但长度不变时不会生效，但更大的问题是——整个 `toolbarActions` 在每次 `selectedChapterIds` 变化时重建，导致 `handleBatchExportConfirm` 等回调函数引用变化，进一步传播重渲染。

改进：将必要的回调用 `useCallback` 包裹，或者让 `useMemo` 依赖 `selectedChapterIds` 而不是只依赖 `.length`。

### 3. 在 `ChapterInfoTable` 级别添加 `React.memo` 

当前只有 `ChapterTableSection` 被 `memo` 包裹，但 `ChapterInfoTable` 是普通函数。由于 `WorkspaceChaptersTab` 的 `activeTabKey` 切换会导致 `ChapterInfoTable` 卸载/挂载（由 `activeTabKey === 'arrange'` 条件决定），这个影响不大。但加上 memo 总归能避免一些不必要的重渲染。

**效果**：小幅减少不必要的重渲染。

## 底层重构方案（中长期，根本解决）

### 1. 虚拟滚动替代分页（核心方案）

引入 `@tanstack/react-virtual` 或 `react-virtuoso`，用虚拟滚动替代 Antd Table 的分页：

- 保留全量列表的浏览体验（无需翻页）
- 仅渲染可视区域的约 20-30 行
- 需要自定义表格渲染（不再使用 Antd `Table` 的 `columns` 配置方式，而是用原生 `<table>` + 虚拟行）

**预估工作量**：
- 安装依赖
- 重构 `ChapterTableSection` 或新建 `VirtualChapterTable` 组件
- 保留列头、排序、选中功能
- 保留行内操作按钮

### 2. 使用 `React.memo` / `useMemo` 精细化控制重渲染链

- 对 `ChapterInfoTable` 包裹 `React.memo`
- 对 `WorkspaceDashboardTab`、`WorkspaceDictionaryTab` 等非 chapters 的 tab 都包裹 `React.memo`
- 确保所有 `onXxx` 回调函数在父组件（App.tsx）中的 `useCallback` 依赖正确
- 分拆大型 `useMemo`（如 `toolbarActions`）为多个独立 memo

### 3. 章节数据分页/增量加载（服务端配合）

当前 `api.getChapters(workspaceId)` 一次性返回全部章节。服务端可以支持分页参数 `?offset=0&limit=100`，客户端在翻页/虚拟滚动滚动到边界时懒加载更多章节。

- 客户端维护章节 Map 而非全量数组
- 搜索功能改为服务端搜索（或客户端正则搜索在已加载的章节子集上）
- 选中功能仍然可以跨分页（基于 id 集合）

### 4. 减少轮询频率 + 校验版本号避免无意义刷新

当前 `usePollingTask` 在 `WorkspaceChaptersTab` 中 5 秒轮询。已经在 `App.tsx:827` 做了版本号跳过逻辑（`getProjectResourceVersions`），但该版本校验发生在调用链中——轮询仍会执行 HTTP 请求。

改进方案：
- 将轮询间隔从 5 秒延长到 15-30 秒
- 或者转移到顶层（App.tsx）统一轮询版本号，仅在有变化时下发新数据
- 或者使用 SSE（Server-Sent Events）推送变更通知

### 5. 使用 `useDeferredValue` 降低筛选时卡顿

当前已在 `searchText` 上使用 `useDeferredValue`，这是一个好做法。可以进一步：
- 在快速连续点击选择行时，使用 `useOptimistic` 或延迟选中状态更新
- 将选中状态更新与渲染分离

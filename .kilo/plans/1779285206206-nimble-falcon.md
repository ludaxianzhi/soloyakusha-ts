# 日志系统改造方案

## 概述

对前后端日志系统进行以下改造：
1. 后端：将日志管理从 EventBus 中解耦为独立的 `LogService` 模块
2. 前端：运行日志改为 SSE 实时流 + 终端风格滚动展示
3. 前端：LLM 请求历史改为单列列表 + 搜索 + 弹窗详情

---

## 1. 后端改造：提取日志服务为独立模块

### 1.1 新增 `src/webui/services/log-service.ts`

将 `EventBus` 中与日志管理相关的逻辑提取到独立 `LogService` 类：

- `addLog(level, message, workspaceId): LogEntry` — 添加日志、触发 SSE 事件
- `getLogs(workspaceId?): LogEntry[]`
- `getLogDigest(workspaceId?): LogDigest`
- `getLogPage(options): LogPage`
- `getLogSession(): LogSession`
- `formatLogExport(format, workspaceId?): { content, contentType, fileName }`
- `clearLogs(): void`
- 内部维护：`logs: LogEntry[]`（上限 500 条，超出裁剪为最近 300 条）、`logIdCounter`、`logSession`

### 1.2 修改 `src/webui/services/event-bus.ts`

- EventBus 不再直接管理日志数据，改为注入 `LogService` 引用
- `addLog` 方法改为委托 `logService.addLog()` 并 emit `log` 事件
- 移除 `getLogs`、`getLogDigest`、`getLogPage`、`getLogSession`、`formatLogExport`、`clearLogs` 等日志管理方法
- 保留 SSE 订阅/发布核心逻辑

### 1.3 修改 `src/webui/routes/events.ts`

- `createEventsRoute` 改为依赖 `LogService` 而非 `EventBus`（日志相关路由）
- SSE 流路由仍依赖 EventBus

### 1.4 修改 `src/webui/app.ts`

- 实例化 `LogService` 并注入 EventBus
- 将 LogService 实例传给 `createEventsRoute`

### 1.5 新增 `responseBody` + `reasoning` 字段到 `LlmRequestHistorySummaryItem`

当前 `LlmRequestHistorySummaryItem` 缺 `responseBody` 和 `reasoning`，需补充以支持详情展示（前端已经用到）。

## 2. 前端改造：运行日志 — 实时流 + 终端风格

### 2.1 新增 `src/webui/client/src/app/log-service.ts`

封装前端日志 API，提供与后端 LogService 对接的方法：
- `getLogsSummary()` → 复用现有 API
- `getLogs(pageParams)` → 复用现有 API（历史查询兜底）
- `clearLogs()` → 复用现有 API
- `downloadLogs(format)` → 复用现有 API
- `getLogStream(onLog)` — 连接到 SSE 事件流监听 `log` 事件（可选：通过 `useEventStream` hook 实现）

### 2.2 修改 `RuntimeLogsPanel`（两个文件）

**`ActivityCenterPage.tsx`** 和 **`ActivityCenterDrawer.tsx`** 中的 `RuntimeLogsPanel` 都需要改造：

#### 数据获取方式变更
- 移除 `usePollingTask` 轮询
- 改为通过 `useEventStream` 的 `onLog` 回调接收实时日志
- 保留首次加载时通过 `api.getLogs()` 获取已有历史日志
- 日志列表上限设为 500 条（新日志到达时如果超出则裁剪）
- 移除分页/加载更多逻辑（改为无限滚动+上限裁剪）

#### 展示方式变更
- 使用 `pre` + 等宽字体（复用现有 `.mono-block` 样式或 `.detail-text-block`）
- 容器设置固定高度 + `overflow-y: auto`
- 新日志到达时自动滚动到底部（如果用户已手动上滚则暂停自动滚动）
- 每条日志以 `[HH:MM:SS] [LEVEL] message` 格式显示，颜色使用 `logColor`
- 不再需要"详情"按钮（内容直接可见）

#### 保留的交互
- 清空、导出按钮
- Session 标签（从首次加载获取）
- 总条数显示

### 2.3 样式调整

在 `styles.css` 中新增 `.runtime-log-stream` 相关样式：
- 等宽字体、深色背景（类似终端）
- 日志行间距、颜色
- 自动滚动行为
- `.log-line-error`、`.log-line-warning`、`.log-line-info`、`.log-line-success` 颜色类

## 3. 前端改造：LLM 请求历史 — 单列列表 + 搜索 + 弹窗详情

### 3.1 修改 `RequestHistoryPanel`（`ActivityCenterPage.tsx`）

#### 布局变更
- 移除左右分栏布局（`.history-panel`、`.history-list-pane`、`.history-detail-pane`）
- 改为单一列表 + 弹出式 `Modal` 显示详情
- 移除 `HistoryDetailPanel` 组件（其内容移至 `Modal` 内）

#### 新增搜索功能
- 列表上方添加 `Input.Search` 控件
- 搜索过滤条件：`modelName`、`meta.label`（任务）、`source`、`meta.stage`
- 搜索使用 `useMemo` 在前端过滤 `items` 列表
- 保留底部"加载更多"按钮（分页加载）

#### 列表项展示字段
每条请求项显示：
- 状态标签（OK/ERR，使用 `Tag`）
- 任务标签（`meta.label`，紫色 Tag）
- 模型名称（`modelName`，蓝色 Tag）
- 来源（`source` Tag）
- 阶段（`meta.stage`，如 `stage 1`）
- Token 统计（`totalTokens`，蓝色 Tag）
- 持续时间（`durationSeconds`，例如 `12.345s`）
- TTFT：当前数据模型无此字段，暂不显示（或在备注中说明可扩展添加）
- TPS：计算 `statistics.totalTokens / durationSeconds`，显示为绿色 Tag，格式 `12.3 t/s`
- 错误信息（如果有，红色文本截断）
- 时间戳

#### 详情弹窗
- 使用 `Modal`，尺寸 `width: 960`
- 内部使用 `Collapse`（与当前 `HistoryDetailPanel` 相同）
- 保留导出、删除按钮
- 加载中显示 Spin

### 3.2 移除冗余代码

- 从 `ActivityCenterPage.tsx` 移除不再使用的组件：
  - `HistoryDetailPanel`（逻辑合并到 Modal 内）
  - `JsonCodeBlock` 和 `tokenizeJson`（可能被 `DetailTextBlock` 替代或保留）
- 删除 `styles.css` 中 `.history-panel` 两栏布局相关样式
- 保留 `.json-code-block` 和 `.detail-text-block` 样式（弹窗内仍使用）

## 5. 实施步骤

1. 后端：创建 `LogService`，从 EventBus 提取日志逻辑
2. 后端：修改 EventBus、routes/events.ts、app.ts 接入 LogService
3. 前端：新建 `log-service.ts`（前端日志 API 封装）
4. 前端：改造 `RuntimeLogsPanel`（两个文件）为 SSE 实时流 + 终端风格
5. 前端：添加终端日志 CSS 样式
6. 前端：重构 `RequestHistoryPanel` 为单列列表 + 搜索 + Modal 详情
7. 前端：清理废弃的 CSS 样式
8. 验证：检查 lint、类型检查

# 方案：LLM 请求即时中断与实时输出监听

## 1. 问题分析

### 现状

当前中断翻译/校对流程时：
- 仅将 `running` 状态的步骤条目回退为 `queued`（`requeueRunningWorkItems`）
- 翻译循环/校对 worker 检查 `processingToken` 或 `task.abortRequested` **仅在**当前区块处理完成后才停止
- LLM HTTP 请求一旦发出就无人能打断，必须等待超时或自然完成

### 根因

- `ChatClient.singleTurnRequest()` 签名中没有 `AbortSignal`
- `fetchWithTimeout()` 使用内部私有 `AbortController`，无法从外部取消
- `collectJsonSse()` 的 reader 循环没有绑定外部中止信号
- 各 LLM 客户端实现（OpenAI、Anthropic、Gemini）均未暴露取消能力
- 翻译循环/校对 worker 没有与 LLM 请求关联的中止控制器

## 2. 修改范围和分层方案

### 2.1 LLM 层：贯穿 AbortSignal

#### `src/llm/types.ts`
- `ChatRequestOptions` 新增 `signal?: AbortSignal`

#### `src/llm/utils.ts`
- `fetchWithTimeout` 改为接受外部 `AbortSignal`，使用 `AbortController.any()`（Node 20+/Bun）或组合信号，使外部 abort 和内部超时可同时生效
- `collectJsonSse` 在循环中检查外部 signal 的 `aborted` 状态，及时 break 和释放 reader

#### `src/llm/base.ts` 及所有子类
- `singleTurnRequest` / `singleTurnResponse` / `conversationResponse` 将 `options.signal` 逐层传递给 `fetchWithTimeout` 和 `collectJsonSse`

#### 涉及文件
| 文件 | 修改内容 |
|---|---|
| `src/llm/types.ts` | ChatRequestOptions 加 `signal?: AbortSignal` |
| `src/llm/utils.ts` | fetchWithTimeout 加外部 signal 参数；collectJsonSse 加 signal 检查 |
| `src/llm/base.ts` | 签名无变化（options 已透传） |
| `src/llm/openai-chat-client.ts` | conversationResponse 中将 signal 传入 fetchWithTimeout |
| `src/llm/anthropic-chat-client.ts` | 同上 |
| `src/llm/gemini-chat-client.ts` | 同上 |
| `src/llm/fallback-chat-client.ts` | 回退切换时传播 signal |
| `src/llm/tool-loop-chat-client.ts` | 内部请求传播 signal |

### 2.2 处理器层：暴露 AbortSignal

#### `TranslationProcessorRequest` / `ProofreadProcessorRequest`
- 新增 `signal?: AbortSignal`

#### 对应处理器
- `DefaultTranslationProcessor`、`StyleTransferTranslationProcessor` 将 signal 传入 `ChatRequestOptions`
- `MultiStageProofreadProcessor`、`SingleStepProofreadProcessor`、`ConsistencyCheckProofreadProcessor` ：
  - 将 signal 传入 `ChatRequestOptions`
  - 在多轮迭代（reviewIterations）中检查 `signal.aborted`，提前抛 `AbortError`

#### 涉及文件
| 文件 | 修改内容 |
|---|---|
| `src/project/processing/translation-processor.ts` | `TranslationProcessorRequest` 加 `signal?` |
| `src/project/processing/proofread-processor.ts` | `ProofreadProcessorRequest` 加 `signal?`；处理器中读 signal 并传入 options |
| `src/project/processing/default-translation-processor.ts` | 传 signal |
| `src/project/processing/style-transfer-translation-processor.ts` | 传 signal，各阶段间检查 abort |

### 2.3 Pipeline/Project 层：维护 AbortController

#### `TranslationProject`
- 新增 `private abortControllers = new Map<string, AbortController>()`
- `abortTranslation()` 触发时，对当前 runId 对应的 controller 调用 `.abort()`
- 每次 `startTranslation()` 时创建新 `AbortController` 并关联到 `currentRunId`
- 新增 `getAbortSignal(): AbortSignal` 供外部获取

#### `translation-project-lifecycle.ts`
- 无变更（中止逻辑已在 `TranslationProject` 中）

#### 涉及文件
| 文件 | 修改内容 |
|---|---|
| `src/project/pipeline/translation-project.ts` | 新增 abortControllers 管理；`abortTranslation` 触发 abort；`startTranslation` 创建 controller |

### 2.4 项目服务层：将 AbortSignal 注入处理循环

#### 翻译循环（`startTranslationLoop`）
- 在工作项处理流程中：
  - 每次 `takeNextPendingItem()` 后创建 `AbortController`
  - 将 `controller.signal` 注入 `processor.processWorkItem()` 的 options 中
  - 当 `state.processingToken !== token` 被检测到时（即 abort 发生后），`controller.abort()`
  - `processWorkItem` 的 catch 中区分 `AbortError`（不记录为失败）vs 真实错误

#### 校对 worker（`runProofreadTask`）
- 同样在每个 batch 处理前关联 `AbortController`
- 当 `taskToken !== state.processingToken` 或 `task.abortRequested` 时 abort 当前请求
- `forceAbortProofread` 也需要触发 controller（当前仅改状态，不够）
- 新增 `projectAbortControllers` 映射，或直接在 `WorkspaceRuntimeState` 中维护当前 abort controller

#### 涉及文件
| 文件 | 修改内容 |
|---|---|
| `src/webui/services/project-service.ts` | 翻译循环中创建/触发 abort controller；校对 worker 中创建/触发 |

### 2.5 辅助改动

#### 自定义 `AbortError`
- 在 `src/llm/utils.ts` 中加 `AbortError` 类型（或复用 `DOMException`），方便调用方 catch

#### `retryAsync` 中止感知
- `retryAsync` 在重试间隔 `sleep` 期间检查外部 signal
- 如果 signal 已 abort，不再重试，直接抛 `AbortError`

#### 涉及文件
| 文件 | 修改内容 |
|---|---|
| `src/llm/utils.ts` | 加 `AbortError`；`retryAsync` 的 sleep 中检查 signal |

## 3. 数据流（以翻译中止为例）

```
用户点击"中止翻译"
  → onProjectCommand('abort')
    → projectService.abortTranslation()
      → project.abortTranslation('webui_abort_requested')
        → 1. projectState.lifecycle.status = 'aborted'
        → 2. requeueRunningWorkItems()  // DB 层面回退
        → 3. abortControllers.get(currentRunId)?.abort()  // 新加
          → 正在进行的 LLM 请求收到 signal，fetch 抛出 AbortError
            → collectJsonSse reader 停止读取
            → fetch 抛错
          → 处理器 catch 到 AbortError
            → 不 submit 结果（或 submit 失败标记）
          → worker 中的 try-catch:
            → processError 是 AbortError → 不记录失败日志
            → activeProcessingCount 递减，循环正常退出
```

## 4. 实时 LLM 输出监听的扩展路径（不需在此轮实现）

### 当前已有的基础设施

- `RequestObserver.onRequestProgress` 已在所有 LLM 客户端中实现，可接收 `completionTextDelta`
- `EventBus` 支持自定义事件类型
- SSE 路由 `/api/events` 已存在

### 未来实现思路

1. 在 `TranslationProcessorFactory.createProcessor()` 或 `createProcessorForProject()` 中创建 `RequestObserver`，将 `completionTextDelta` 通过 EventBus emit
2. 新增 SSE 事件类型（如 `llmProgress`），包含 `chapterId`、`fragmentIndex`、`textDelta` 等字段
3. 前端通过 EventSource 接收并渲染到对应区块的 UI 中
4. 无需修改 LLM 客户端代码

### 与本方案的协同

- LLM AbortSignal 机制和 RequestObserver 机制相互独立，不会冲突
- AbortSignal 的中止可通过 `RequestObserver.onRequestError` 通知前端

## 6. 需要决策的问题

1. 翻译/校对被强行打断时，正在处理的片段结果是否丢弃？
   - 推荐：丢弃（不调用 `submitWorkResult`），因为 LLM 输出可能不完整。
2. abort 后是否需要等待所有并发 worker 退出再返回响应？
   - 推荐：`abortTranslation()` 继续保持异步返回（先返回状态），不阻塞等待。
3. 是否需要对 `stopTranslation({ mode: 'immediate' })` 也启用立即打断？
   - 推荐：是，和 abort 使用同一套 AbortController 机制。

# 翻译项目持久化层重构方案

## 现状问题分析

### 问题 1：章节/片段元数据不在 SQLite 中独立存储
- `fragments.source_json` 存储完整的源文行文本 JSON 数组
- `loadChapterDescriptorsSync()` 读取时通过 `SUM(json_array_length(source_json))` 计算行数，**每次调用都要解析所有片段的全部源文 JSON**
- `getChapterDescriptor()` 在 `TranslationProjectWorkspace` 中有两个路径：`getChapterDescriptors()` 走存储层聚合；`getRequiredChapterDescriptor()` 走内存对象遍历 — 两者效率都不高

### 问题 2：`fragment` 与 `fragment_lines` 表职责混淆
**当前 `fragments` 表存储了本应由行表管理的逐行数据：**
```
fragments:   fragment_lines:
  source_json      →  (缺失，应该在这)
  metadata_json    →  (缺失，应该在这)
  target_groups_json → (缺失，应该在这)
  aux_data_json         translation (唯一的内容)
  hash
  ...
```
- 要获取一行原文，必须解析整个 `source_json` JSON 数组
- 要获取一行 metadata，必须解析整个 `metadata_json` JSON 数组
- `fragment_lines` 只存 `translation`，职责单一但不对称

### 问题 3：行级 metadata 没有独立管理
- `TranslationUnitMetadata`、`targetGroups` 作为 JSON 数组嵌入 `fragments` 表
- 无法单独查询/更新某行的 metadata，必须整体读写

### 问题 4：缺少 `extend` 和 `comment` 预留字段
- 行表没有扩展字段，不利于未来功能扩展

---

## 重构方案

### 核心原则
1. **无须向后兼容**：旧工作区文件直接作废，不写迁移脚本
2. **SQLite schema 按职责拆分**：fragment 只存元数据，line 数据全归 `fragment_lines`
3. **章节级聚合数据预计算**：`chapters` 表直接存储 `fragment_count`、`source_line_count`、`translated_line_count`
4. **内存模型不改**：`FragmentEntry` 等 domain types 基本不变（hydrator 层做映射），减少调用方改造成本

---

### 新 Schema

```sql
CREATE TABLE project_metadata (
  key TEXT PRIMARY KEY,
  value_json TEXT NOT NULL
);

CREATE TABLE chapters (
  chapter_id INTEGER PRIMARY KEY,
  file_path TEXT NOT NULL,
  fragment_count INTEGER NOT NULL DEFAULT 0,
  source_line_count INTEGER NOT NULL DEFAULT 0,
  translated_line_count INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE fragments (
  chapter_id INTEGER NOT NULL,
  fragment_index INTEGER NOT NULL,
  hash TEXT NOT NULL,
  line_count INTEGER NOT NULL DEFAULT 0,
  aux_data_json TEXT NOT NULL DEFAULT '{}',
  PRIMARY KEY (chapter_id, fragment_index),
  FOREIGN KEY (chapter_id) REFERENCES chapters(chapter_id) ON DELETE CASCADE
);

CREATE TABLE fragment_lines (
  chapter_id INTEGER NOT NULL,
  fragment_index INTEGER NOT NULL,
  line_index INTEGER NOT NULL,
  source TEXT NOT NULL,
  translation TEXT NOT NULL DEFAULT '',
  metadata_json TEXT,
  target_groups_json TEXT,
  extend TEXT NOT NULL DEFAULT '',
  comment TEXT NOT NULL DEFAULT '',
  PRIMARY KEY (chapter_id, fragment_index, line_index),
  FOREIGN KEY (chapter_id, fragment_index)
    REFERENCES fragments(chapter_id, fragment_index) ON DELETE CASCADE
);

CREATE TABLE pipeline_step_states (
  chapter_id INTEGER NOT NULL,
  fragment_index INTEGER NOT NULL,
  step_id TEXT NOT NULL,
  state_json TEXT NOT NULL,
  PRIMARY KEY (chapter_id, fragment_index, step_id),
  FOREIGN KEY (chapter_id, fragment_index)
    REFERENCES fragments(chapter_id, fragment_index) ON DELETE CASCADE
);

CREATE INDEX idx_fragments_hash ON fragments(hash);
CREATE INDEX idx_pipeline_step_states_step
  ON pipeline_step_states(step_id, chapter_id, fragment_index);
```

### 变更前后对比

| 数据 | 旧位置 | 新位置 |
|------|--------|--------|
| source 文本 | `fragments.source_json` | `fragment_lines.source` |
| metadata | `fragments.metadata_json` | `fragment_lines.metadata_json` |
| targetGroups | `fragments.target_groups_json` | `fragment_lines.target_groups_json` |
| auxData | `fragments.aux_data_json` | `fragments.aux_data_json` (不变) |
| hash | `fragments.hash` | `fragments.hash` (不变) |
| lineCount | (无) | `fragments.line_count` (新增) |
| translation | `fragment_lines.translation` | `fragment_lines.translation` (不变) |
| fragmentCount | (聚合计算) | `chapters.fragment_count` (预存) |
| sourceLineCount | (聚合计算) | `chapters.source_line_count` (预存) |
| translatedLineCount | (聚合计算) | `chapters.translated_line_count` (预存) |
| extend | (无) | `fragment_lines.extend` (新增) |
| comment | (无) | `fragment_lines.comment` (新增) |

### 影响文件与修改计划

#### 1. `src/project/types.ts` — 类型调整
- `FragmentMeta`：保留原结构（hydrator 从 line table 聚合），不需要大改
- `ChapterEntry`：保留 `fragments: FragmentEntry[]`，不变
- `FragmentEntry`：保留 `source`, `translation`, `meta`, `pipelineStates`, `hash` 字段，不变
  - **hydrator 层**负责从 `fragment_lines` 行数据重建这些字段

#### 2. `src/project/storage/sqlite-project-storage.ts` — **核心重构**

##### 新增/修改的类型定义
```typescript
type FragmentRow = {
  chapter_id: number;
  fragment_index: number;
  hash: string;
  line_count: number;
  aux_data_json: string;
};

type FragmentLineRow = {
  chapter_id: number;
  fragment_index: number;
  line_index: number;
  source: string;
  translation: string;
  metadata_json: string | null;
  target_groups_json: string | null;
  extend: string;
  comment: string;
};
```

##### Schema 初始化 (`openDatabase`)
- 替换 `CREATE TABLE fragments`，移除 `source_json`, `metadata_json`, `target_groups_json`，添加 `line_count`
- 替换 `CREATE TABLE fragment_lines`，添加 `source`, `metadata_json`, `target_groups_json`, `extend`, `comment`
- 替换 `CREATE TABLE chapters`，添加 `fragment_count`, `source_line_count`, `translated_line_count`
- 移除 `idx_fragment_lines_lookup`（主键索引已覆盖）
- 清理 `aux_data_json` 列的旧 ALTER TABLE 迁移代码

##### `replaceChapter(db, chapter)`
- 写入 `chapters` 行（含聚合计数：`fragment_count`, `source_line_count`, `translated_line_count`）
- 写入 `fragments` 行（不含行文本，增加 `line_count`）
- 写入 `fragment_lines` 行（含 `source`, `translation`, `metadata_json`, `target_groups_json`, `extend`, `comment`）
- 写入 `pipeline_step_states`

##### `hydrateChapter()` / `hydrateFragment()`
- 从 `fragment_lines` 读取完整行数据（source + translation + metadata + targetGroups）
- 重构 `FragmentEntry.source` 和 `FragmentEntry.translation`（`TextFragment`）
- 重构 `FragmentEntry.meta.metadataList` 和 `meta.targetGroups`

##### `replaceFragmentTranslations(db, ...)`
- 改为写回完整的 `fragment_lines` 行（source + translation + metadata + targetGroups）
- 写入时 `ON CONFLICT` 更新 translation 列
- **更新 `chapters.translated_line_count`**（重新统计或递增递减）

##### `updateFragmentTranslation()` → 改为 `updateFragmentTranslation()`
- 更新 `fragment_lines.translation` 和 `chapters.translated_line_count`

##### `updateTranslatedLine()`
- 保持不变（单行 UPDATE fragment_lines SET translation）

##### `loadChapterDescriptorsSync()`
- **直接从 `chapters` 表读取聚合数据**，不再 `SUM(json_array_length(...))`
- 性能从 `O(总行数)` 降为 `O(章节数)`

##### `loadChapterSync()`
- 从 `fragments` 读取 hash/line_count/aux_data
- 从 `fragment_lines` 读取完整行数据
- 调用新的 `hydrateChapter()` / `hydrateFragment()` 重建 `FragmentEntry`

##### `loadChapterIndexSync()`
- 从 `fragments` 获取 `hash`（移除 `source_json` 依赖，表结构轻量化）

##### 新增方法
- `updateChaptersLineCount(chapterId, sourceLineCount, translatedLineCount)` — 当翻译状态变更时更新 chapters 聚合计数
- `getFragmentLine(chapterId, fragmentIndex, lineIndex)` — 按需读取单行数据
- `updateFragmentLineExtend(chapterId, fragmentIndex, lineIndex, extend)` — 更新 extend 字段
- `updateFragmentLineComment(chapterId, fragmentIndex, lineIndex, comment)` — 更新 comment 字段

#### 3. `src/project/document/translation-document-manager.ts` — 适配存储层

##### `createFragmentEntry(fragmentUnits)`
- 不变（从 TranslationUnit[] 构建 FragmentEntry，逻辑不变）

##### `saveChapter(chapter)`
- 写入前计算 `chapters` 的聚合计数
- 计算 `fragment_count = chapter.fragments.length`
- 计算 `source_line_count = sum(fragment.source.lines.length)`
- 计算 `translated_line_count = sum(fragment.translation.lines.filter(l => l.length > 0).length)`
- 这些值写入 chapters 表

##### `loadChapters()`
- 现有流程不变（先查索引，不存在则加载并初始化）

##### `updateTranslation()`
- 更新 `fragment_lines.translation`
- 更新 `chapters.translated_line_count`

##### `updateTranslatedLine()`
- 现有单行 UPDATE 保持不变
- 额外更新 `chapters.translated_line_count`

##### `updateStepStateAndTranslation()`
- 保持不变（存储层会处理多表写）

##### `mergeFragmentAuxData()` / `replaceFragmentAuxData()`
- 不变（aux_data 仍在 `fragments` 表）

##### `getChapterTranslationUnits()` / `mergeChapterTranslationUnits()`
- 不变（从 `FragmentEntry` 聚合，只关心 in-memory 数据）

##### `flattenChapterLines()`
- 不变（从 `FragmentEntry` 读取）

#### 4. `src/project/pipeline/translation-project-workspace.ts` — 优化描述符读取

##### `getChapterDescriptors()`
- 当前调用 `documentManager.getAllChapterDescriptors()` → `storage.loadChapterDescriptorsSync()`
- 重构后 `loadChapterDescriptorsSync()` 从 `chapters` 表直接读取
- 性能从 O(n) JSON 解析 → O(n) 简单列读取

##### `getRequiredChapterDescriptor(chapterId)`
- 当前遍历内存中 chapter 的所有 fragment 计算行数
- 可以改为优先从 `chapters` 表的聚合数据读取（引入 `getAllChapterDescriptors()` 方式）
- 但保留内存遍历作为兜底（当数据尚未持久化时）

#### 5. 测试文件调整

##### `src/project/pipeline/project.test.ts`
- 大部分是集成测试，存储层变更后工作流不变，只需调整 setup 时对 DB 格式的期望
- 如测试中有直接读取 DB 的断言，需要适配新 schema

##### `src/project/document/translation-document-manager.test.ts`
- 如果存在，需要更新直接操作存储层的测试

##### `src/project/context/chapter-translation-editor.test.ts`
- `createChapterEntryForTest()` — 不涉及 SQLite，纯构建内存对象，**不需要修改**

##### `src/project/context/context-network-storage.test.ts`
- 不涉及 fragments 表，不变

### 注意事项

1. **`extend` 字段用途**：设计为自由 JSON 字符串（`TEXT`），消费者可以序列化任意结构进去。当前不定义具体 schema，产品需求到来时再扩充。
2. **`comment` 字段用途**：人工审校/编辑器场景的文本块行级备注。纯文本，不限制长度。
3. **事务完整性**：`replaceChapter()` 在一个事务中完成 chapters + fragments + fragment_lines + pipeline_step_states 的写入，不会出现部分写入。
4. **`loadChapterSync()` 与 `saveChapter()` 配对使用**：没有独立的 fragment_lines 粒度读取，要么整章读，要么整章写。单行翻译更新走专用的 `updateTranslatedLine()` 方法。
5. **并发写队列**：`enqueueWrite()` 机制保持不变，保证写操作串行化。
6. **hash 计算不变**：`Bun.hash(fragmentToText(source)).toString(16)` 保持不变，因为 source 内容不变。
